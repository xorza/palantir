# Vizia — reference notes for Palantir

Vizia is George Atkinson's (geom3trik) declarative Rust UI toolkit: a `View` trait stored in a flat `Entity → Box<dyn ViewHandler>` map, layout delegated to **morphorm**, styling via a parsed **CSS subset** (selectors, pseudo-classes, animations), and a state-binding story that originally borrowed Druid's `Lens` + `Data` derive but has since migrated wholesale to Leptos-style **signals** (see `tmp/vizia/MIGRATION_LENS_TO_SIGNALS.md`). It's the most useful real-world test case for morphorm — Palantir's `Sizing` story is morphorm's `Units` minus three units, so reading vizia is reading the future of "what if we kept all four."

All paths under `tmp/vizia/crates/vizia_core/src/` unless noted.

## 1. `View` / `Handle` / Modifiers — declarative composition

`View` (`view.rs:82`) is small:

```rust
pub trait View: 'static + Sized {
    fn build<F: FnOnce(&mut Context)>(self, cx: &mut Context, content: F) -> Handle<Self> { … }
    fn element(&self) -> Option<&'static str> { None }
    fn event(&mut self, cx: &mut EventContext, event: &mut Event) {}
    fn draw(&self, cx: &mut DrawContext, canvas: &Canvas) { /* draws bg/border/text */ }
    fn accessibility(&self, cx: &mut AccessContext, node: &mut AccessNode) {}
}
```

`build` (`view.rs:113`) is the load-bearing default method. It allocates an `Entity` from `cx.entity_manager`, threads it into `cx.tree` under the current parent, registers cache/style/access entries, boxes `self` into `cx.views: HashMap<Entity, Box<dyn ViewHandler>>` (`view.rs:128`), then runs the user-supplied `content` closure with `cx.with_current(entity, …)` so child `View::new(cx, …)` calls auto-parent. Identity is per-call: the entity is fresh on every build, and persistence happens because `Application::run` only calls the root build closure *once* — the tree is retained, not rebuilt per frame. Compare Floem's `ViewId` allocation; same shape, different storage (slotmap vs `HashMap`).

`Handle<'a, V>` (`view/handle.rs:9`) is `{ current: Entity, entity: Entity, p: PhantomData<V>, cx: &'a mut Context }` returned from `build`. It exists for one reason: to host the **modifier** chain (`some_view.background_color(Color::red()).width(Pixels(100.0))`). Each modifier is generated by a `modifier!` macro (e.g. `modifiers/layout.rs:33` for `layout_type`) that takes either a literal value *or a signal* and stamps it into the appropriate `Style` field on `cx.style` for the handle's entity, raising the right `SystemFlags` (`RELAYOUT`, `REDRAW`, `RESTYLE`). This is the entire styling-as-API: `Handle` is a builder over the same property maps the CSS parser writes into.

Two-tier composition: `View::build` opens a scope where children parent under self; modifiers chain after. `cx.with_current(entity, content)` (`view.rs:159`) is the analogue of egui's `ui.with_layer` or our own `Ui::container` push/pop — Vizia uses a single `current: Entity` field on `Context` instead of a parent stack because the `build` closure form already nests via Rust's call stack.

## 2. `Model` + lens (deprecated) → signals

`Model` (`model.rs:37`) is a per-entity event-handling trait:

```rust
pub trait Model: 'static + Sized {
    fn build(self, cx: &mut Context) -> &Self { … }
    fn event(&mut self, cx: &mut EventContext, event: &mut Event) {}
}
```

`Model::build` boxes `self` into `cx.models: HashMap<Entity, HashMap<TypeId, Box<dyn ModelData>>>` keyed by the *current* entity (`model.rs:66`). At lookup time, `DataContext::try_data::<T>` walks `entity.parent_iter` to find the nearest ancestor that owns a `T` (`view/handle.rs:23`). That parent-walk is what made the *original* `Lens` system work: `AppData::count` was a derive-generated `Lens<AppData, i32>`, and a `Label::new(cx, AppData::count)` would do `cx.try_data::<AppData>()` → run the lens → subscribe the label's entity to that field. Druid's design ported wholesale, with the same `Data::same` change-detection predicate and the same `derive(Lens)` proc-macro.

`MIGRATION_LENS_TO_SIGNALS.md:1` documents the rip-and-replace: model fields go from `T` to `Signal<T>`, writes go from `self.count += 1` to `self.count.update(|c| *c += 1)`, and views take signal handles directly. The reactive runtime now lives in a separate `vizia_reactive` crate (Leptos-style: `Runtime`, `Scope`, `Signal`, `Effect`, `Memo`). `Binding::new(cx, signal, |cx| …)` (`binding/binding.rs:46`) is the explicit "rebuild this subtree on signal change" primitive: it allocates an `Entity`, marks it ignored for layout, registers an `UpdaterEffect` whose `on_change` callback inserts the entity id into a thread-local `SIGNAL_REBUILDS: HashSet<Entity>`. The next frame's binding system drains that set and reruns each binding's content closure under the binding's scope — disposing the old children and rebuilding from scratch. That's the unit of granularity: a `Binding` block, not a single label.

The reason for the migration is exactly what Druid hit (and what `references/druid.md §5` enumerates): lenses-into-`Vec` are unsound under reorders, `Data::same` for collections forces `Arc<Vec<T>>` or `im`, derived state needs awkward `LensExt::map` chains. Signals fix all three by replacing structural projection with explicit subscription. The cost: every reactive field is now a `Signal<T>` allocation with a refcell underneath, and the reactive runtime is `!Send` thread-local — same trade-off Floem made.

## 3. Morphorm integration — `Units::{Pixels, Percentage, Stretch, Auto}`

`crates/vizia_core/src/layout/mod.rs:33` re-exports `morphorm::{LayoutType, PositionType, Units}` directly as the public layout vocabulary. There is no Vizia-side wrapper enum. `Style` holds an `AnimatableSet<Units>` per layout property (`width`, `height`, `left`, `right`, `top`, `bottom`, `min_width`, `max_width`, `padding_left`, …, `horizontal_gap`, `vertical_gap`).

`impl Node for Entity` (`layout/node.rs:14`) wires Vizia into morphorm. `Node::Store = Style`, `Node::Tree = Tree<Entity>`, `Node::CacheKey = Entity`, `Node::ChildIter<'t> = MorphormChildIter<'t, Entity>` (which skips `ignored` entities — that's how `Binding` parents are layout-invisible). Each axis getter is the same shape:

```rust
fn width(&self, store: &Self::Store) -> Option<morphorm::Units> {
    store.width.get_resolved(*self, &store.custom_units_props).map(|w| match w {
        Units::Pixels(val) => Units::Pixels(store.logical_to_physical(val)),
        t => t,
    })
}
```

`get_resolved` resolves CSS-cascade + custom-property overrides; `logical_to_physical` applies DPI scale only to absolute pixel units (percentage/stretch/auto are dimensionless). `Node::content_size` (`node.rs:90`) is morphorm's text-measure callback — it pulls the entity's `text_paragraphs` Skia paragraph, calls `paragraph.layout(width)`, and returns `(width, height)` after subtracting padding. This is the path morphorm uses for `Auto`-sized leaves: text intrinsic width *during* the parent's main-axis stretch resolution.

The actual layout call is one line (`systems/layout.rs:19`):

```rust
Entity::root().layout(&mut cx.cache, &cx.tree, &cx.style, &mut SubLayout { … });
```

Then a fixup walk converts morphorm's parent-relative outputs to absolute screen coordinates (`systems/layout.rs:33-90`) and emits `GeoChanged` flags so paint/clip caches invalidate per-axis. Layout runs on the *whole* tree when `SystemFlags::RELAYOUT` is set (`systems/layout.rs:14` has a `TODO: incremental` — they haven't done it yet, six years in). Compare WPF's `MeasureQueue` or Floem's Taffy `mark_dirty`: vizia is the most coarse of the three and it still ships. The reason it works is that morphorm itself is a single recursive pass with a small per-node cache, and trees are O(hundreds), not O(thousands).

The Sizing-unit mapping for Palantir's purposes:
- `Sizing::Fixed(n)` = `Units::Pixels(n)`.
- `Sizing::Hug` = `Units::Auto` *for the size axis*, with the parent's `padding_*` consumed via `Node::padding_*`. (Vizia/morphorm has no `Hug` keyword; `Auto` does double duty for "shrink to content" on size-fields and "defer to parent padding" on edge-fields. See `references/morphorm.md §3`.)
- `Sizing::Fill` = `Units::Stretch(1.0)`. Multi-fill weighting falls out of `Stretch(f32)` factors — something Palantir doesn't currently support but could trivially.

What morphorm gives that Palantir doesn't: per-edge `Stretch` (so a child can centre itself by setting `left: Stretch(1), right: Stretch(1)`), `Percentage` units, `min_width`/`max_width` clamps, `Grid` layout, and `Overlay`. What it costs: more units to teach (`references/morphorm.md` is 99 lines unpacking the four), more `Option<Units>` getters per axis (16+ on `Node`), and the conceptual blur between "I am stretched" and "my left margin is stretched" — both go into the same free-space distribution loop in morphorm.

## 4. CSS subset — `vizia_style`

`crates/vizia_style/` is a hand-written CSS parser built on `cssparser` and `selectors` (the Servo/Stylo crates). The grammar lives in `parser/` (~36 lines main + 263 in `rule.rs`); property parsing in `property.rs`; values in `values/` — about 60 files, one per CSS value type (`color.rs`, `gradient.rs`, `font_weight.rs`, `easing.rs`, `border/`, `transform.rs`, …). Output is a `StyleSheet` of `StyleRule { selectors, declarations }` consumed by `crates/vizia_core/src/systems/style.rs:1` (1475 lines).

Selectors: type (`hstack`, `label`, custom `View::element`), class (`.primary`), id (`#submit`), pseudo-class (`:hover`, `:active`, `:focus`, `:checked`, `:disabled`, `:placeholder-shown`, `:over`, `:focus-visible` — `vizia_style/src/pseudoclass.rs`), descendant/child/sibling combinators. No attribute selectors. The `selectors` crate's matching machinery is reused via `impl SelectorImpl for Selectors` (`vizia_style/src/matching.rs`), so specificity, nesting, `:not`/`:is`/`:where` come for free.

Animations: `transition: background-color 200ms ease-out` is parsed in `values/easing.rs` and `values/duration.rs` and stored on the rule. `animation/animation_builder.rs` and `interpolator.rs` interpolate `AnimatableSet<T>` properties; the `Interpolator` trait is impl'd per value type (`Color`, `Units`, `Transform`, `BoxShadow`). `systems/animation.rs` ticks active animations each frame and writes interpolated values into `Style` with `RESTYLE`/`RELAYOUT` flags. `@keyframes` is supported (`vizia_style/src/rules/keyframes.rs`).

`systems/style.rs:1` does the cascade: a full restyle walks the tree, runs `compute_matched_rules` per entity, builds a sorted-by-specificity rule list, and writes resolved values back into the per-property `AnimatableSet` storage. This is *not* incremental except via the `SystemFlags::RESTYLE` gate — touch one rule, restyle the world. The 1475-line file is mostly the per-property "did this change? raise this flag" plumbing.

The CSS-as-styling bet pays off for theming and skinnability (designers can ship `.css` files), and pays a tax in: a multi-thousand-LOC parser, a 1475-line cascade system, and two sources of truth (modifier API and stylesheet) that have to be kept in sync as new properties are added. Every new style property requires changes in `vizia_style::Property`, the parser, the modifier macro, the cascade, the interpolator, and the morphorm `Node` impl if it's layout-relevant.

## 5. Event propagation

`Event` (`events/event.rs:22`) wraps `message: Option<Box<dyn Any + Send>>` plus `EventMeta { target, origin, propagation, consumed, … }`. `Propagation` (`event.rs:8`) is `Up | Subtree | Direct`. `EventManager::flush_events` (`events/event_manager.rs:39`) drains `cx.event_queue` once per frame, dispatching each event according to its `propagation`:

- `Direct` — only the `target` entity's `event` is called.
- `Up` — `target.parent_iter(&cx.tree)` from leaf to root, with `meta.consumed` shortcircuiting.
- `Subtree` — `TreeIterator::subtree(&tree, target)` pre-order walk.

For each visited entity, the event manager pulls the boxed view out of `cx.views` (`event_manager.rs:86` does the temporary `.remove`/`.insert` dance to satisfy the borrow checker — view receives `&mut self` and `&mut Context` simultaneously), then calls `view.event(&mut EventContext::new(cx), event)`. Models attached to the same entity get the event next, via `cx.models.get_mut(&entity)` and `ModelData::event`. This is how `cx.emit(AppEvent::Increment)` from a button click reaches `impl Model for AppData::event` — the event bubbles up, the nearest ancestor with an `AppData` model handles it.

There's also a separate listener path (`event_manager.rs:77`): per-entity `cx.listeners` closures get the event before view dispatch, used for one-off subscriptions and `keyboard_event_lock_root` modal scopes. Global listeners run before any entity-targeted dispatch (`event_manager.rs:67`).

The whole event system is single-threaded and synchronous. Async results enter via `cx.spawn` → channels → posting `Event::new(msg)` to the queue; the event loop processes the queue between frames. Same shape as Druid's `ExtEventSink`/`Command`, less ceremony.

## 6. Lessons for Palantir

**Validate.**
- **Morphorm is a real test case for the four-units design.** Vizia ships production apps on `Pixels | Percentage | Stretch | Auto`, and `references/morphorm.md §2-3` documents that the model holds up — child-controls-its-own-spacing (centring via `left: Stretch(1), right: Stretch(1)`) is genuinely a nicer factoring than flexbox's parent-side `justify-content`. Palantir's `Sizing::{Fixed, Hug, Fill}` is the same primitive minus `Percentage` and minus per-edge stretch. If we add `Percentage` later, copy the morphorm resolution: `Units::Pixels(n) → n` directly, `Units::Percentage(p) → p/100 * parent_axis`, `Units::Stretch(f) → leftover * f / Σf` after fixed siblings are accounted for. The hard part is teaching the user when to use which; the algorithm is straightforward.
- **`Style` as a typed property bag indexed by entity** is the same shape as our future per-widget persistent-state map. Vizia's `Style.width: AnimatableSet<Units>` (one per layout property) is overkill for a tree we rebuild every frame, but the *animation* hook — interpolating a property over time without rebuilding the widget — is the one place the retained-tree approach is genuinely better than ours. We'll need a small interpolator-on-`Id-Any-map` story when transitions arrive.

**Reject.**
- **Lenses, even after the migration.** Vizia spent years on `derive(Lens)` + `Data::same`, hit the same Druid walls (`Vec` lensing, derived state, async), and ripped it out in favour of signals. The migration guide (`MIGRATION_LENS_TO_SIGNALS.md`) is a confession in markdown form. Palantir's bet — user code reads its own state directly during `record`, no projection, no subscription — bypasses both the lens era and the signal era. Don't build a lens system. Don't build a signal runtime. The user already has Rust closures.
- **CSS as the styling DSL.** The `vizia_style` crate alone is bigger than Palantir's entire codebase, and `systems/style.rs` is 1475 lines just to walk the cascade. Two costs: maintenance (every new property touches six files) and divergence (modifier API and stylesheet drift, and which one wins under animation is non-obvious). Palantir's plan should be a Rust-typed `Style` struct settable per-call in the record pass — `Button::new(id).background(Color::RED).padding(8.0)` — plus optional themes as plain `Style` values composed by the user. Designers who want skinnability get a JSON/RON theme file; we don't ship a CSS parser.
- **Whole-tree relayout on every dirty bit.** Vizia's `systems/layout.rs:14` candidly admits the TODO. They get away with it because morphorm is fast and trees are small. Palantir already rebuilds the tree every frame — we're not worse off. Don't bother with incremental layout until profiling forces it.
- **The `Box<dyn ViewHandler>` retained tree.** `cx.views: HashMap<Entity, Box<dyn ViewHandler>>` is the *exact* persistence shape Palantir's design rejects. Vizia needs it to give views a place to host event handlers and `&mut self` `draw` overrides; we rebuild every frame and dispatch events against last frame's rects, so the box-per-widget allocation is gone. Stay flat.

**Borrow.**
- **Event `Propagation::{Up, Subtree, Direct}`** as the propagation enum once Palantir grows hit-testing and bubbling. `Up` is what 90% of UI events want; `Direct` is for command dispatch; `Subtree` is for "broadcast to children" (theme change, focus revoked). The simpler model is to start with `Up` and `Direct`, add `Subtree` only when a use case demands it.
- **`event.consume()` returning a single bit on the event** rather than a richer "handled / wants more" tri-state. egui's `Response::clicked` is more granular but ours can be `Response { hovered, clicked, /* … */ }` with consume-on-handler responsible for stopping propagation. Vizia's `meta.consumed` shortcircuit in `event_manager.rs:101` is the entire mechanism. Copy that shape.
- **`Handle::modify(|view| …)` escape hatch** (`view/handle.rs:94`) for views that need imperative tweaks after construction. Our equivalent might be returning `&mut Node` from `Button::new(id)`'s builder, but only if a real use case appears.

**Single biggest takeaway.** Vizia is the proof that morphorm's `Pixels | Percentage | Stretch | Auto` is a real layout vocabulary, *and* the proof that retained trees in Rust eventually pay the same lens-or-signals tax Druid and Floem pay. Palantir takes morphorm's idea (four uniform units, child-side spacing authority) without taking the retained-tree commitment that forces lenses or signals into the picture. Read `references/morphorm.md` for the layout algorithm itself; this file is for what *Vizia* adds on top, almost all of which Palantir is choosing not to add.
